Literature Search · Methods Tool

QueryBee Boolean & fielded query builder for any field

Describe a topic, expand each concept into terms (with any free AI assistant, or by mining the literature), and the tool assembles correctly fielded, fully parenthesized search strings for Scopus, Web of Science, Google Scholar, and PubMed in each engine's own syntax. The syntax assembly is deterministic and field-agnostic. Treat it as a starting scaffold for a search, not a replacement for a librarian's strategy or for screening; the literature term suggestions are most reliable in well-indexed, English-language fields. No account, no key, no cost, and everything runs in your browser.

Scopus · TITLE-ABS-KEY Web of Science · TS= Google Scholar · full text Free · works with any chatbot
New here? How to use this tool
1. In step 1, type your topic and set the year range and which databases you want.
2. Either copy the step 2 prompt into any free chatbot and paste its reply into step 3, or click a starter block and edit it. On any block, find terms in literature suggests synonyms ranked by how distinctive they are to your topic.
3. Step 4 shows a ready-to-run query for each database; copy it into that database's advanced search. The review panel flags weak spots in your strategy.
4. In step 6, paste a few papers you already know are relevant to see whether your search would find them, and which block to fix if it would miss one.
5. In step 7, save or share the whole search, or export a record for your methods section.
It is free, needs no account, and your search never leaves your browser. For a fuller explanation, see USER-GUIDE.md in the project folder.

What the tool applies, and what to watch

      
Each concept sits in its own field code and the blocks are joined only with AND, so operator precedence cannot regroup the logic. Keep that habit: precedence rules differ between engines and Scopus changed its order recently, so parenthesize every block rather than trusting defaults. Date and document-type filters are translated per engine: Scopus uses exclusive PUBYEAR > / < and codes such as ar, re; Web of Science uses an inclusive PY=() range and full names such as Article, Review, Proceedings Paper.

      
The two subscription databases will not return the same count. Web of Science TS= also searches Keyword Plus, index terms generated from the references a paper cites, which Scopus TITLE-ABS-KEY has no equivalent for. The gap is why searching both improves recall.

      
Google Scholar is different in kind. It searches full text, not separate title, abstract, and keyword fields, it has no document-type filter, it does not support stem truncation (its asterisk is a whole-word placeholder), and it cuts off queries beyond about 256 characters. The tool therefore removes truncation asterisks from the Scholar query, shows the character count, and warns when a Scholar query runs long. Treat Scholar as fast, free scoping, not the systematic search of record, and trim terms to the core concepts for that panel.

      
PubMed keeps real field tags, so the tool writes each term as term[tiab] for title and abstract, bounds the years with ("YYYY"[dp] : "YYYY"[dp]), and maps document types it recognizes, "Journal Article"[pt] and Review[pt]; conference papers and book chapters have no clean PubMed type and are dropped from that panel. PubMed covers biomedicine and the life sciences only, so it returns little or nothing for topics outside that scope. The thesis, book, and open-access links search full text and are for discovery rather than a reproducible search of record.

How it works. The query syntax for each database is assembled by the tool itself, so the fielded structure is consistent and editable. The AI step only expands a topic into concept terms, and it runs in whatever chatbot you already use, so the tool needs no API key and costs nothing. Multi-word terms are quoted as phrases, and a trailing asterisk truncates word stems in Scopus, Web of Science, and PubMed. Google Scholar does not support stem truncation, so the asterisk is removed from the Scholar query automatically.

    
Hosting it for a class. This is a single static file. Put it on any web host or open it locally and it works for everyone, because it calls no paid service. The only external request is the optional find-terms helper, which mines candidate terms from the OpenAlex scholarly database (free, no account, no key) and falls back to a public dictionary or fails quietly when offline, so the core query builder always works.
